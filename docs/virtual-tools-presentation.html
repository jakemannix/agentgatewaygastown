Part 1: Virtual Tools

Simple 1:1 Overrides

What Are Virtual Tools?

Simple 1:1 mappings that override backend tool properties:

  • Rename — Give agent-friendly names
  • Redescribe — Override the description
  • Reshape — Override input/output schemas
  • Inject — Add default parameter values
  • Hide — Remove fields from schema

All without touching the backend tool!

Virtual Tool Definition

pub struct VirtualToolDef {
    pub name: String,           // Override: exposed name
    pub source: ToolSource,     // Reference to backend tool
    pub description: Option<String>,    // Override: description
    pub input_schema: Option<Value>,    // Override: input schema
    pub defaults: HashMap<String, Value>, // Inject: default values
    pub hide_fields: Vec<String>,       // Hide: remove from schema
    pub output_schema: Option<OutputSchema>, // Override: output
    pub version: Option<String>,        // Semantic version
    pub metadata: HashMap<String, Value>, // Arbitrary metadata
}

Example: Weather Tool Override

{
  "name": "get_weather",
  "source": { "target": "weather-service", "tool": "fetch_weather" },
  "description": "Get current weather for a city",
  "defaults": {
    "api_key": "${WEATHER_API_KEY}",
    "units": "metric"
  },
  "hideFields": ["debug_mode", "raw_output"],
  "outputSchema": {
    "properties": {
      "temperature": { "sourceField": "$.data.current.temp_f" },
      "conditions": { "sourceField": "$.data.current.condition.text" }
    }
  },
  "version": "2.1.0"
}

Override: Rename & Redescribe

{
  "name": "get_customer_emails",
  "source": {
    "target": "salesforce",
    "tool": "query_contacts"
  },
  "description": "Get email addresses for customers at a company"
}

Backend: salesforce.query_contacts
Agent sees: get_customer_emails

Override: Input Schema

Simplify what the agent needs to provide:

{
  "inputSchema": {
    "type": "object",
    "properties": {
      "company_name": {
        "type": "string",
        "description": "Company name to search for"
      }
    },
    "required": ["company_name"]
  }
}

Backend expects complex SOQL → Agent provides simple company_name

Inject: Default Values

Automatically inject parameters:

{
  "defaults": {
    "api_key": "${WEATHER_API_KEY}",
    "timeout": 30,
    "units": "metric"
  }
}

Environment variable substitution: ${VAR_NAME}
Agent doesn't need to provide these values

Hide: Remove Fields from Schema

{
  "hideFields": [
    "debug_mode",
    "raw_output",
    "internal_trace_id",
    "legacy_format"
  ]
}

These fields are removed from the exposed schema
Agent never sees them, can't accidentally set them

Override: Output Schema (JSONPath)

Transform output with JSONPath extraction:

{
  "outputSchema": {
    "type": "object",
    "properties": {
      "temperature": {
        "type": "number",
        "sourceField": "$.data.current.temp_f"
      },
      "conditions": {
        "type": "string",
        "sourceField": "$.data.current.condition.text"
      }
    }
  }
}

Backend returns 50+ fields → Agent sees 2 fields

Part 2: Tool Registry

Centralized Discovery & Management

What Is The Tool Registry?

A centralized catalog that manages virtual tools:

  • Discovery — Find available tools
  • Versioning — Semantic versions for tools
  • Hot-Reload — Update without restart
  • Multiple Sources — File or HTTP

Registry Configuration

# From local file (with hot-reload on change)
registry:
  source: file:///etc/agentgateway/tools.json
  refreshInterval: 30s

# From HTTP endpoint
registry:
  source: http://registry.corp.com/api/v1/tools
  refreshInterval: 5m
  auth:
    bearer: ${REGISTRY_TOKEN}

Registry JSON Structure

{
  "schemaVersion": "1.0",
  "tools": [
    {
      "name": "get_weather",
      "source": { "target": "weather", "tool": "fetch_weather" },
      "description": "Get current weather",
      "version": "2.1.0",
      "defaults": { "units": "metric" },
      "hideFields": ["debug_mode"],
      "outputSchema": { ... }
    },
    {
      "name": "search_docs",
      "source": { "target": "confluence", "tool": "search" },
      "version": "1.3.0",
      ...
    }
  ]
}

Versioning

Tools have semantic versions:

{
  "name": "get_weather",
  "version": "2.1.0",
  "metadata": {
    "owner": "weather-team",
    "dataClassification": "public",
    "deprecatedAt": null
  }
}

Track changes, detect drift, manage deprecation

Hot-Reload Mechanism

pub struct RegistryStore {
    // Atomic registry replacement via ArcSwap
    current: Arc<ArcSwap<Option<CompiledRegistry>>>,
}

impl RegistryStore {
    // Readers see updates without blocking
    pub fn update(&self, registry: Registry) {
        let compiled = CompiledRegistry::compile(registry);
        self.current.store(Arc::new(Some(compiled)));
    }
}

File source: Watcher triggers on file change
HTTP source: Background refresh at interval

Request Flow

ListTools:

Agent requests tools → Gateway queries backends →
Registry applies overrides → Returns virtual tool list

CallTool:

Agent calls get_weather → Registry resolves to weather/fetch_weather →
Injects defaults → Calls backend → Transforms output → Returns

Part 3: Tool Algebra

Advanced N:1 Compositions

The Problem

Virtual tools are 1:1 mappings.

But what about composing multiple tools?

  • Search 3 sources in parallel, aggregate results
  • Chain tools: search → fetch → summarize
  • Route to different tools based on input
  • Retry failed operations with backoff

Today: imperative glue code in every agent

The Vision

What if scatter_gather, pipeline, circuit_breaker, and retry were first-class primitives you could compose declaratively?

Inspired by Apache Camel's Enterprise Integration Patterns

The Analogy

Apache Camel MCP Tool Algebra
Endpoint (HTTP, JMS) MCP Server/Tool
Message Tool Input/Output (JSON)
Route Tool Pipeline/Workflow
Processor Tool Result Transformer

Pattern Catalog

PatternTypePurpose
PipelineN:1Sequential: output feeds next input
Scatter-GatherN:1Parallel: query multiple, aggregate
Router1:NDispatch based on input content
Filter1:1Pass/block based on predicate
NormalizerN:1Unify heterogeneous outputs
EnricherN:1Augment with additional data
Circuit Breaker1:1Fail fast on repeated failures
SagaN:1Distributed transactions

Pattern: Pipeline

Chain tools where output feeds next input:

// research_pipeline: (query: string) -> Summary
pipeline(
    web_search,      // string -> SearchResult[]
    extract_urls,    // SearchResult[] -> URL[]
    web_fetch,       // URL[] -> Document[]
    summarize        // Document[] -> Summary
)
query → [search] → [extract] → [fetch] → [summarize] → Summary

Pattern: Scatter-Gather

Query multiple sources in parallel, aggregate results:

// knowledge_search: (query: string) -> RankedAnswer
scatter_gather({
    targets: [web_search, arxiv_search, internal_docs],
    aggregate: [flatten, dedupe, sort_by_relevance, limit(10)],
    timeout: 5s
})
        ┌→ web_search ────┐
query ──┼→ arxiv_search ──┼→ Aggregate → Results
        └→ internal_docs ─┘

Pattern: Content-Based Router

Dispatch to different tools based on input:

// analyze_document: (file: File) -> ExtractedText
route(input) {
    when(input.type == "pdf")  -> pdf_tool.extract
    when(input.type == "xlsx") -> xlsx_tool.read
    when(input.type == "docx") -> docx_tool.parse
    otherwise                  -> generic_extractor
}

Pattern: Normalizer

Convert heterogeneous outputs to common schema:

// unified_search: (query: string) -> UnifiedDocument[]
normalize(
    sources: {
        gdrive: gdrive_search,      // {name, link, mimeType}
        slack: slack_search,        // {title, url, ts, channel}
        confluence: confluence_search // {page, href, space}
    },
    to_schema: UnifiedDocument      // {title, content, source, url}
)

Resilience Patterns

Circuit Breaker

circuit_breaker({
    tool: external_api,
    failure_threshold: 5,
    reset_timeout: 30s,
    fallback: local_cache
})

Retry with Backoff

retry({
    tool: flaky_service,
    max_attempts: 3,
    backoff: exponential(
        base: 100ms,
        max: 5s
    )
})

Pattern: Saga

Multi-step transactions with compensation on failure:

// order_workflow: (order: Order) -> Success | RolledBack
saga({
    steps: [
        { action: reserve_inventory, compensate: release_inventory },
        { action: charge_payment, compensate: refund_payment },
        { action: ship_order, compensate: cancel_shipment }
    ]
})

If step 3 fails → run compensate[2], compensate[1], compensate[0]

Composing Patterns

// ResearchPipeline: (topic: string) -> Report
workflow ResearchPipeline(topic) {
    // Scatter-gather from multiple sources
    sources = scatter_gather({
        targets: [web_search, arxiv_search, internal_docs],
        timeout: 10s
    })

    // Normalize to common schema
    normalized = normalize(sources, to: ResearchDocument)

    // Filter and enrich
    relevant = normalized
        | filter(doc => doc.relevance > 0.7)
        | enrich(doc => { ...doc, citations: get_citations(doc.id) })

    return aggregate(relevant, strategy: synthesize_report)
}

MCP-Specific Patterns

Beyond Enterprise Integration

AI/LLM-Specific Patterns

  • Schema Mediator
    LLM-assisted transformation between incompatible schemas
  • Capability Router
    Route based on tool capabilities discovered at runtime
  • Semantic Deduplicator
    Merge by semantic similarity, not exact match
  • Confidence Aggregator
    Weight results by source reliability

Confidence Aggregator Example

// drug_interaction_check: (drug_a, drug_b) -> SafetyAssessment
confidence_aggregate({
    sources: [
        { tool: fda_database,     weight: 0.95 },  // authoritative
        { tool: pubmed_search,    weight: 0.80 },  // peer-reviewed
        { tool: drugs_com,        weight: 0.70 },  // curated
        { tool: web_search,       weight: 0.30 },  // unverified
        { tool: llm_knowledge,    weight: 0.20 }   // may hallucinate
    ],
    strategy: weighted_consensus
})
